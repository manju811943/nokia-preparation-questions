49. What are the ES6 features :
	=> ES6 (ECMAScript 2015) introduced many modern JavaScript features that make code cleaner, shorter, and easier to maintain.

	1. Let and Const :
		Purpose
		=> Block-scoped variables (safer than var).
			Example
				let count = 10;
				count = 20;   // allowed

				const PI = 3.14;
				// PI = 3.15; ❌ not allowed

		=>	Real-time Use Case
				In a React or Angular component, constants like API URLs should not change.
				const API_URL = "https://api.company.com/users";

	2. Arrow Functions
			Purpose
				=> Shorter function syntax and automatic this binding.
				Example
					const add = (a, b) => a + b;
					console.log(add(5, 3)); // 8

			=> Real-time Use Case :
				Used heavily in React event handlers or array operations.
	

	3. Template Literals
			Purpose
				String interpolation using backticks.
			Example
			let name = "Roy";
			let message = `Hello ${name}`;
			console.log(message);

			=> Real-time Use Case :
			 	Generating dynamic UI messages or API URLs
					let userId = 10;
					let url = `https://api.company.com/users/${userId}`;

	4. Destructuring
			Purpose
				Extract values from objects/arrays easily.
			Example
			const user = {
			  name: "Roy",
			  age: 30
			};
			const {name, age} = user;
			console.log(name);

		=>	Real-time Use Case :
				Very common in React props or API responses
	5. Spread Operator (...)
			Purpose
				Copy or merge arrays/objects.
			Example
				const arr1 = [1,2];
				const arr2 = [...arr1, 3,4];
				console.log(arr2);

	6. Default Parameters
			Purpose
				Provide default values to function parameters.
			Example
			function greet(name = "Guest"){

		=>	Real-time Use Case
				Handling optional parameters in API functions

	7. Classes
			Purpose

			Example

51. NgRx Store vs BehaviorSubject (Angular State Management) ?
		| Feature      | NgRx Store              | BehaviorSubject Service    |
		| ------------ | ----------------------- | -------------------------- |
		| State type   | Global state management | Local/shared service state |
		| Architecture | Redux pattern           | Simple RxJS                |
		| Complexity   | High                    | Low                        |
		| Boilerplate  | More files              | Minimal code               |
		| Best for     | Large applications      | Small/medium features      |

	=> BehaviorSubject is a simple RxJS-based state sharing mechanism typically used inside Angular services for small or localized state management. 
	   NgRx Store, on the other hand, follows the Redux pattern and provides a centralized global store with actions, reducers, effects, and selectors. 
       NgRx is better suited for large enterprise applications where predictable state management, debugging, and scalability are important.

53. what are injection token ?

		})

		})

			@Inject(API_URL) private apiUrl: string
			) {}
				getClaims() {
				return this.http.get(`${this.apiUrl}/claims`);
			}
		}

🔹 Why StringBuffer is Thread-Safe?

				Prevents race conditions
				Guarantees data consistency
		So, when multiple threads try to modify the same StringBuffer, access is serialized.

		🔑 Key point:

94.How do you avoid performance issues caused by synchronization?

1️⃣ Understand Why Synchronization Hurts Performance
Synchronization can cause:
	Thread contention (threads waiting on locks)
	Context switching overhead
	Reduced parallelism
	Risk of deadlocks
	=> Goal: Minimize locking, not eliminate thread safety

2️⃣ Reduce Lock Scope (Most Important)
❌ Bad (large synchronized block):
		public synchronized void process() {
		    validate();
		    calculate();
		    save();
		}	
✅ Good (lock only critical section):
		public void process() {
		    validate();
		    synchronized (this) {
		        calculate();
		    }
		    save();
		}
✔ Less time holding lock
✔ Higher throughput

3️⃣ Prefer Concurrent Collections
Map<String, Integer> map = new ConcurrentHashMap<>();
Why it’s faster
	Lock striping
	Non-blocking reads
	Fine-grained locking

4️⃣ Use Atomic Classes Instead of synchronized
❌ Bad
	synchronized(this) {
	    count++;
	}

✅ Good(lock free, Highly scalable)
AtomicInteger count = new AtomicInteger();
count.incrementAndGet();

5️⃣ Use Executor Framework Instead of Manual Synchronization
✅ Good
	ExecutorService executor = Executors.newFixedThreadPool(10);
	executor.submit(() -> process());

		✔ Thread reuse
		✔ Less contention
		✔ Controlled concurrency
6️⃣ Avoid Nested Locks (Deadlock Prevention)
❌ Bad
	synchronized(lock1) {
	    synchronized(lock2) {
	    }
	}

✅ Lock ordering or single lock strategy
✔ Avoid deadlocks
✔ Predictable performance

123. 🔄 Sync vs Async (Synchronous vs Asynchronous) :

	=> In a synchronous call, the client sends a request and waits for the response before continuing.
			Example:
			A standard REST API call:
				Service A → calls Service B
				Service A waits
				Service B responds
				Service A continues
			Characteristics:
				Blocking
				Immediate response required
				Simple request–response pattern
				Tight coupling (Service A depends on Service B being available)
				
	=> In an asynchronous model, the client sends a message and does not wait for an immediate response.
			Example:
			Using a message broker like Apache Kafka or RabbitMQ:
				Service A publishes a message
				Service A continues immediately
				Service B processes the message later
			Characteristics:
				Non-blocking
				Event-driven
				Loose coupling
				Better resilience
				
	==> when to use Kafka/RabbitMQ(Async) :
		1. Decoupling services:
			Services should not fail just because another service is down.
			Example:
				Order Service publishes "OrderCreated"
				Inventory, Email, Billing consume it independently.
				
		2.High throughput / scalability
			Apache Kafka is especially good for:
				Millions of events

				Log aggregation

126. 🔹 What are equals() and hashCode()?

	=> Both methods come from java.lang.Object and are heavily used by hash-based collections like HashMap, HashSet, Hashtable.
	
	| Method       | Purpose                                                         | return type |
	| ------------ | --------------------------------------------------------------- |	---------- |
	| `equals()`   | Checks **logical equality** (content comparison)                |	boolean     |
	| `hashCode()` | Helps Java **quickly locate objects** in hash-based collections |	int         |
	
	👉 Rule:
	If two objects are equal according to equals(), they must have the same hashCode().
	
	=> 🔹 How hashCode() works INTERNALLY 🧠
			When you add an object to a HashMap / HashSet:
				Java calls hashCode()
				Hash code is converted into a bucket index
				Object is stored in that bucket
			
	=> 🔹 How equals() works INTERNALLY 🧠
			Used only when needed:
				Java first checks hashCode()
				If hashCodes are different → objects are NOT equal
				If hashCodes are same → Java calls equals()
			
		=> if (obj1.hashCode() == obj2.hashCode()) {
				obj1.equals(obj2);
			}
			
	🔹 Contract Between equals() and hashCode() 📜
			Java enforces this contract:
				1️. If a.equals(b) is true
				➡️ a.hashCode() == b.hashCode() must be true

				2️. If hashCode() is same
				➡️ equals() may be true or false

				❌ Breaking this contract causes bugs in HashMap / HashSet
				
	==> ✔️ Statement 1 : If equals() returns true, then hashCode() must be the same
	==> ✔️ Statement 1 : If two objects have the same hashCode, equals() may return true or false

129. what is dead lock and How to AVOID Deadlock ✅ (MOST IMPORTANT)

	=> A deadlock occurs when two or more threads are permanently blocked, each waiting for a resource held by the other
	
		👉 None of the threads can proceed
		👉 Application appears hung forever
		
	=> 🔹 How to AVOID Deadlock ✅ (MOST IMPORTANT)
	
	1. Use tryLock() (Timeout-based locking) :
	
		Lock lock1 = new ReentrantLock();
		Lock lock2 = new ReentrantLock();

		if (lock1.tryLock()) {
			try {
				if (lock2.tryLock()) {
					try {
						// work
					} finally {
						lock2.unlock();
					}
				}
			} finally {
				lock1.unlock();
			}
		}

	2. Minimize Lock Scope :
		synchronized(this) {
			// only critical code
		}
	
	3. Avoid Nested Locks (if possible) :
		synchronized(a) {
			synchronized(b) { }
		}

	4. Use Higher-Level Concurrency APIs :
		=> ExecutorService
		=> ConcurrentHashMap
		=> BlockingQueue

131. 🔥 synchronized vs ReentrantLock :

		=> synchronized is a Java keyword that provides implicit locking using the JVM’s intrinsic (monitor) lock.
		
		Eg:	synchronized void increment() {

			}
			
		=> Lock is automatically acquired & released
		=> Simple and safe
		=> Managed entirely by JVM

		=> ReentrantLock is a class from java.util.concurrent.locks package that provides explicit locking with advanced features.
		
		Eg : Lock lock = new ReentrantLock();

			lock.lock();
			try {
				count++;
			} finally {
				lock.unlock();
			}
			
			=> Locking is manual
			=> More control
			=> More powerful

138. 🧵 Thread Life Cycle in Java :

	=> In Java, a thread goes through several well-defined states during its lifetime. 
		These states are defined in the Thread.State enum.
	
	1. NEW :
		=> A thread is in the NEW state after it is created but before it is started.
		=> It has not yet begun execution.

		=> Thread t = new Thread();
			✔ Thread object created
			❌ start() not called yet
		
	2. RUNNABLE :
		=> After calling start(), the thread moves to RUNNABLE.
		=> It is ready to run and waiting for CPU time.
		=> It may be running or ready to run (Java does not distinguish between ready and running).

		=> t.start();
			✔ Eligible for execution
			✔ Controlled by the thread scheduler
		
	3. BLOCKED :
		=> A thread enters BLOCKED state when it is waiting to acquire a monitor lock.
		=> Happens when trying to enter a synchronized block/method that is already locked by another thread.

			synchronized(obj) {
			   // critical section
			}

			✔ Waiting for lock
			❌ Cannot proceed until lock is available
			
			
	4. WAITING :
		=> The thread waits indefinitely until another thread performs a specific action.
		=> Triggered by methods like:

			wait()
			join()
			LockSupport.park()

			✔ No CPU usage
			✔ Waiting for notification
			
	5. TIMED_WAITING :
		=> The thread waits for a specified amount of time.

		=> Triggered by:
			sleep(1000)
			wait(1000)
			join(1000)

			✔ Waiting with timeout
			✔ Automatically returns after time expires
			
	6. TERMINATED
		=> The thread enters TERMINATED after execution completes.
		=> It cannot be restarted.

			=> // run() method finishes

			✔ Thread is dead
			❌ Cannot call start() again
			
	🎯 Important Notes :
		=> A thread can move between RUNNABLE ↔ BLOCKED/WAITING/TIMED_WAITING multiple times.
		=> Once in TERMINATED, it cannot restart.
		=> The thread scheduler controls execution order.
		
		
139. 🔥 Difference Between volatile and synchronized in Java ?

	✅  volatile :
			What it does:
				=> Ensures visibility of variable changes across threads.
				=> Prevents instruction reordering.
				=> Does NOT provide locking.
				=> Does NOT guarantee atomicity.

			Example:
				=> volatile boolean flag = false;

				If Thread A changes flag = true,
				Thread B immediately sees the updated value.

			❌ But This Is Not Safe:
				volatile int count = 0;
				count++;   // Not atomic!
				=> Still causes race conditions.
				
	✅ synchronized :
			What it does:
			=> Provides mutual exclusion (locking).
			=> Ensures only one thread executes the block/method at a time.
			=> Guarantees visibility + atomicity.

			Example:
				synchronized void increment() {
					count++;
				}

				✔ Thread-safe
				✔ No race condition
				

140. If I want to write a thread safe code, what all things I should consider ?

	=> Writing thread-safe code is not just about adding synchronized. It’s about designing your code 
	   so that multiple threads can access shared data safely without causing race conditions, 
	   visibility issues, or deadlocks.
	   
	 ✅ Identify Shared Mutable State (Most Important) :
			Ask yourself:
				=> Is this variable accessed by multiple threads?
				=> Can it change (mutable)?
				=> Is it read + written by different threads?
				
			👉 If yes, it needs protection.
				eg: int count = 0;   // Shared + Mutable = Risky
				
				
	✅ Prefer Immutability (Best Strategy) :
			=> Immutable objects are automatically thread-safe.

			Eg:	final class User {
					private final String name;
					private final int age;

					public User(String name, int age) {
						this.name = name;
						this.age = age;
					}
				}
				
				✔ No setters
				✔ All fields final
				✔ No synchronization needed

			💡 Best practice: Make classes immutable whenever possible.
			
			
	✅ Use Proper Synchronization for Shared Data :
			=> If multiple operations must happen together, use:
			✔ synchronized
					synchronized(this) {
						count++;
					}

			✔ Locks
					Lock lock = new ReentrantLock();
					lock.lock();
					try {
						count++;
					} finally {
						lock.unlock();
					}

			✔ Atomic Classes
					AtomicInteger count = new AtomicInteger(0);
					count.incrementAndGet();
				
	✅ Use volatile Only for Visibility (Not Locking) :
			=> Use volatile when:
				=> Only simple read/write
				=> No compound operations
			
			Good example:
			eg: volatile boolean shutdown;

			Bad example:
			eg:	volatile int count;
				count++;   // Still unsafe
				
	✅ Avoid Deadlocks :
			Deadlock happens when:
				=> Thread A → holds Lock1 → waiting for Lock2
				=> Thread B → holds Lock2 → waiting for Lock1

			Prevention:
				=> Always acquire locks in the same order
				=> Keep synchronized blocks small
				=> Avoid nested locks when possible
				
	✅ Minimize Scope of Synchronization :
			Bad:
			Eg:	public synchronized void method() {
					// 100 lines of code
				}

			Better:
			Eg:	public void method() {
					// non-critical code

					synchronized(this) {
						// only critical section
					}
				}

			✔ Improves performance
			✔ Reduces contention
			
	✅ Prefer High-Level Concurrency Utilities :
			Instead of manual threads, use:
				=> ExecutorService
				=> ConcurrentHashMap
				=> BlockingQueue
				=> AtomicInteger
				=> CompletableFuture
			These are safer and more scalable.
			
			
141. visibility vs atomcity vs ordering ?

	🔍 Visibility
		👉 Meaning:
			When one thread updates a variable, will other threads see the updated value immediately?

		❌ Without Visibility Guarantee :
				=> Each thread may cache variables locally.
					eg:	boolean flag = false;

						Thread A:
						flag = true;

						Thread B:
						while (!flag) { }   // may loop forever


				=> Thread B may never see the update.

		✅ How to Guarantee Visibility
				=> volatile
				=> synchronized
				=> Lock
				=> Atomic classes

					eg: volatile boolean flag;

				Now changes become visible to all threads immediately.
				
	⚛ Atomicity
			👉 Meaning:
					=> An operation happens completely or not at all (indivisible).
					=> No other thread can see it halfway.

			❌ Not Atomic
				count++;   // NOT atomic

				This is actually:
					=> Read
					=> Increment
					=> Write

			Two threads can interfere → wrong result.

			✅ Atomic Operations

				=> synchronized
				=> Lock
				=> AtomicInteger.incrementAndGet()

				Eg: AtomicInteger count = new AtomicInteger();
					count.incrementAndGet(); 
					
	🔄 Ordering
		👉 Meaning:
				=> The execution order of instructions may be rearranged (reordered) by:
					1. Compiler
					2. JVM
					3. CPU

		For performance optimization:
		
				=> Example of Reordering
						int a = 1;
						int b = 2;

				=> The CPU might execute:
						b = 2;
						a = 1;

				=> Single thread → no problem
				=> Multiple threads → can cause serious bugs
				

	=> 🧠 Summary Table :
	
		| Concept    | What It Solves                  | Example Tool                    |
		| ---------- | ------------------------------- | ------------------------------- |
		| Visibility | Other threads see latest value  | `volatile`                      |
		| Atomicity  | Operation cannot be interrupted | `synchronized`, `AtomicInteger` |
		| Ordering   | Prevent instruction reordering  | `volatile`, `synchronized`      |
		

142. 🧵 What is a BlockingQueue in Java?

	=> A BlockingQueue is a thread-safe queue used in producer–consumer scenarios where:
			🔹 Producers add elements
			🔹 Consumers remove elements
			🔹 Threads automatically wait (block) when needed
		=> It is part of Java’s concurrency utilities (java.util.concurrent).

	✅ Why "Blocking"?
			Because operations block the thread when:
				1. Queue is empty → take() waits
				2. Queue is full → put() waits

			=> So you don’t need to manually use:
				1. wait()
				2. notify()
				3. synchronized
				
		Eg: BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(2);

				// Producer
				new Thread(() -> {
					try {
						queue.put(1);
						queue.put(2);
						queue.put(3); // waits (queue full) -> Queue size = 2
					} catch (InterruptedException e) {
						e.printStackTrace();
					}
				}).start();

				// Consumer
				new Thread(() -> {
					try {
						Thread.sleep(2000);
						System.out.println(queue.take()); // removes 1 
					} catch (Exception e) {}
				}).start();

157. What is SSO?

	Single Sign-On (SSO) lets a user log in once and access multiple applications without logging in again.
	
	Eg : Login to Company Portal → Access Email, HR, Jira, GitLab automatically.
	
	🔄 SSO Login Flow (OAuth 2.0 / OpenID Connect) :
	✅ Step-by-Step Flow
	
	1. User accesses Application
		User → App
			User opens the app (e.g., HR portal).
		
	2. App redirects to Identity Provider
		App → IdP (Login Page)
			Because the user is not authenticated, the app redirects to the IdP.
		
	3. User authenticates at IdP
		User → IdP
			User enters credentials (password, OTP, biometrics).
			✔ Authentication happens only at IdP
			
	4. IdP issues tokens
		IdP generates:
			ID Token (who the user is)
			Access Token (permissions)
			
	5. Tokens sent back to Application
		IdP → App (via redirect)
		
	6. App validates token
			Signature
			Expiry
			Issuer
		✔ If valid → user is logged in
		
	7. User accesses other apps (SSO magic ✨)
		User → Another App → IdP → Already logged in ✔
			No login screen again
			
	🎯 Interview One-Liner (Must Remember):
	
			=> In SSO, authentication is handled by a central Identity Provider. Once authenticated, 
			   the IdP issues a token that allows the user to access multiple applications without logging in again.

161. If two Employee objects have the same ID & name, will a Set allow duplicates?
	
	👉 It depends on whether equals() and hashCode() are overridden correctly.
	
	🔴 Case 1: equals() & hashCode() NOT overridden ❌
			class Employee {
				int id;
				String name;
			}

			Set<Employee> set = new HashSet<>();
			set.add(new Employee(1, "John"));
			set.add(new Employee(1, "John"));
			
	❌ Result:
			Duplicates ARE allowed
			Because:
				Default equals() → compares memory references
				Different objects → different references
				
				
	🟢 Case 2: equals() & hashCode() overridden correctly ✅
			class Employee {
				int id;
				String name;

				@Override
				public boolean equals(Object o) {
					if (this == o) return true;
					if (!(o instanceof Employee)) return false;
					Employee e = (Employee) o;
					return id == e.id && Objects.equals(name, e.name);
				}

				@Override
				public int hashCode() {
					return Objects.hash(id, name);
				}
			}

			Set<Employee> set = new HashSet<>();
			set.add(new Employee(1, "John"));
			set.add(new Employee(1, "John"));

			✅ Result:
				Duplicate is NOT allowed
				Second add() is ignored

			Every Java object has:
				=> Monitor (intrinsic lock)
				=> Used for thread synchronization
				=> synchronized keyword locks an object, not a thread

		=> What These Methods Do :
			| Method        | Purpose                       |
			| ------------- | ----------------------------- |
			| `wait()`      | Releases lock and waits       |
			| `notify()`    | Wakes **one** waiting thread  |
			| `notifyAll()` | Wakes **all** waiting threads |

187. What Are Isolation Levels?
	=> Isolation levels define how and when the changes made by one transaction become visible to other concurrent transactions.

	The 4 SQL Isolation Levels (Low → High) :
	1.READ UNCOMMITTED (Lowest)
		👉 A transaction can read uncommitted data from another transaction.
			❌ Dirty Read
			❌ Non-repeatable Read
			❌ Phantom Read
		📌 Rarely used (unsafe)

	2. READ COMMITTED (Most Common – Default in many DBs)
		👉 Can read only committed data.
			✅ Dirty Read prevented
			❌ Non-repeatable Read
			❌ Phantom Read

	3.REPEATABLE READ
		👉 Same row read multiple times gives same value.
			✅ Dirty Read prevented
			✅ Non-repeatable Read prevented
			❌ Phantom Read (mostly*)

	4. SERIALIZABLE (Highest)
		👉 Transactions behave as if executed one after another.
			✅ Dirty Read
			✅ Non-repeatable Read
			✅ Phantom Read

	🔥 Real-World Example (Bank Account)
			Scenario:
			Two users reading & updating same account balance.
				READ COMMITTED → balance may change between reads
				REPEATABLE READ → balance remains same
				SERIALIZABLE → strict locking, safest

198. How ConcurrentHashMap Works (Before Java 8) and after java 8 ?

		🔹 Java 7 and Below
				It used Segment-based locking.
					Map divided into multiple segments
					Each segment had its own lock
					Only one thread per segment could write
				📌 This reduced locking compared to Hashtable, but still had segment-level lock.

		🚀 What Changed in Java 8?
	 		1. No More Segments
					Java 8 removed Segment class completely.
				Instead, it uses:
					CAS (Compare-And-Swap)
					Synchronized block at bucket level
					Node-based locking

			2. Uses CAS for Updates
				For insertion:
					If bucket is empty → use CAS (no lock)
					If bucket not empty → synchronize only that bucket
				Much faster than segment locking.

			3. Tree Structure for High Collisions
					Just like HashMap in Java 8:
					If bucket size > 8 → convert linked list into Red-Black Tree

			4.Better Parallelism
					Multiple threads can operate on different buckets simultaneously.
					Reads are mostly non-blocking.

		| Feature            | Java 7        | Java 8                      |
		| ------------------ | ------------- | --------------------------- |
		| Locking            | Segment-level | Bucket-level                |
		| Segments           | Yes           | ❌ Removed                   |
		| Collision Handling | LinkedList    | LinkedList + Red-Black Tree |
		| Performance        | Good          | 🚀 Better                   |



199. what is optmistic and pesimistic lock in hibernate ?
	🔐 Locking in Hibernate
			Locking is used to handle concurrent updates on the same database record and maintain data consistency.
			
	
	Hibernate supports two types of locking:
		Optimistic Locking
		Pessimistic Locking
		
	1. Optimistic Locking
	
		🔹 What it is
				Assumes conflicts are rare
				No database lock while reading
				Checks data consistency at update time
				
	Eg:	@Entity
		class Order {

			@Id
			private Long id;

			@Version
			private int version;

			private String status;
		}
		
	Flow:
		Two users read the same record (version = 1)
			User A updates → version becomes 2
			User B updates → ❌ OptimisticLockException

		🔹 When to use
			Read-heavy applications
			Less chance of concurrent updates
			High performance required

		📌 Interview line:
			=> Optimistic locking uses a version column and checks for conflicts during update, without locking the database row.
			
	2. Pessimistic Locking
			What it is
				Assumes conflicts are common
				Locks the database row immediately
				Other transactions must wait
				
		🔹 When to use
				High contention data
				Financial transactions
				Inventory / seat booking systems
				
			eg:	entityManager.find(
				   Order.class,
				   id,
				   LockModeType.PESSIMISTIC_WRITE
				);

			📌 Interview line:
				Pessimistic locking locks the database row so that no other transaction can't modify it until the lock is released.
				
				
				
	| Feature           | Optimistic Locking      | Pessimistic Locking |
	| ----------------- | ----------------------- | ------------------- |
	| Lock Type         | No DB lock              | DB row lock         |
	| Conflict Handling | At commit time          | At read time        |
	| Performance       | High                    | Lower               |
	| Waiting           | No waiting              | Other threads wait  |
	| Failure           | OptimisticLockException | Lock timeout / wait |

					Priority order (high → low)
						Command-line arguments
						JVM system properties
						Environment variables
						application-{profile}.yml
						application.yml / properties

				✅ Result: App starts using highest-priority value
				
		Case 2: Conflicting Beans (Startup FAILURE) : two beans of same types

						   @Qualifier

				What happens?
					Spring Boot uses @ConditionalOnMissingBean
					Your bean wins
					App starts fine

				✅ No failure (by design)
				
		Case 4: Invalid / incompatible configuration values
				Eg : server.port=ABC

		| --------------------- | ----------------------- |

208. What are Virtual Threads (Project Loom, Java 21) and how do they improve backend scalability?

	What are Virtual Threads?
		Virtual Threads are lightweight threads introduced in Java (Project Loom) that dramatically reduce the cost of creating and managing threads.
		Introduced as preview in Java 19
		Stabilized in Java 21
		Implemented by the JVM (not the OS)
		
		👉 In simple words:
			Virtual threads allow you to run millions of concurrent tasks using the same blocking-style code.

	Platform Thread vs Virtual Thread (Simple Comparison)
		| Feature                | Platform Thread  | Virtual Thread            |
		| ---------------------- | ---------------- | ------------------------- |
		| Backed by OS thread    | Yes              | ❌ No                      |
		| Creation cost          | Expensive        | Very cheap                |
		| Memory per thread      | ~1–2 MB          | Few KB                    |
		| Max concurrent threads | Thousands        | Millions                  |
		| Blocking call impact   | Blocks OS thread | ❌ Doesn’t block OS thread |

	=> In traditional systems, each request blocks an OS thread while waiting for DB or external APIs. 
	   Virtual threads are lightweight and when they wait, they release the underlying OS thread. 
	   This allows the system to handle thousands of concurrent I/O requests using simple blocking code without thread exhaustion

211. How do CompletableFuture and Structured Concurrency enhance async REST handling ?
	=> 🧵 First: The Problem in Async REST
				In your Fraud Service, suppose for one claim you must:
					Call Policy Service
					Call Customer Service
					Call GAF (external)
					Fetch DB history

					If you do it sequentially:
						Call A (500ms)
						Call B (500ms)
						Call C (500ms)
							Total = 1500ms
				=> But these calls are independent.
				
				We want:
					Call A, B, C together
					Total ≈ 500ms
					
	1. CompletableFuture – Improves Parallel REST Calls
		What It Does :
			CompletableFuture allows:
				Running tasks asynchronously
				Combining results
				Handling errors
				Executing in parallel

		🧠 Simple Example (Your Fraud Case) :  all below are run asyncrnously or parallely and at end they combine and process

		eg:	CompletableFuture<Policy> policyFuture =
			    CompletableFuture.supplyAsync(() -> policyClient.getPolicy());
			
			CompletableFuture<Customer> customerFuture =
			    CompletableFuture.supplyAsync(() -> customerClient.getCustomer());
			
			CompletableFuture<FraudScore> gafFuture =
			    CompletableFuture.supplyAsync(() -> gafClient.getScore());
			
			CompletableFuture.allOf(policyFuture, customerFuture, gafFuture).join();
			
			Policy policy = policyFuture.join();
			Customer customer = customerFuture.join();
			FraudScore score = gafFuture.join();

	How It Enhances Async REST Handling :
		✔ Reduces overall latency
		✔ Improves response time
		✔ Uses thread pool efficiently
		✔ Allows combining multiple REST responses

	🚨 Problem With CompletableFuture
			When logic becomes complex:
				Hard to manage cancellation
				Hard to handle partial failure
				If one fails → others still run
				Thread leaks possible
			It becomes messy in large systems.

so, in Java 21 inriduced called Structured Concurrency – Cleaner & Safer Async (project Loom)

	🧠 Idea Behind Structured Concurrency
			If a parent request fails:
				All child tasks should stop
			
			If one child fails:
				Cancel others
				Everything is managed in one block.

	eg:		try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
			
			    Future<Policy> policy = scope.fork(() -> policyClient.getPolicy());
			    Future<Customer> customer = scope.fork(() -> customerClient.getCustomer());
			    Future<FraudScore> gaf = scope.fork(() -> gafClient.getScore());
			
			    scope.join();           // Wait for all
			    scope.throwIfFailed();  // If any failed
			
			    return combine(policy.resultNow(),
			                   customer.resultNow(),
			                   gaf.resultNow());
			}


		✅ Why This Is Better
		
		✔ Automatic cancellation
		✔ Cleaner error handling
		✔ No thread leaks
		✔ Easier debugging
		✔ Safer parallel execution

	| Feature            | CompletableFuture | Structured Concurrency |
	| ------------------ | ----------------- | ---------------------- |
	| Parallel execution | ✅                 | ✅                      |
	| Error handling     | Complex           | Clean                  |
	| Auto cancellation  | ❌ Manual          | ✅ Automatic            |
	| Readability        | Medium            | High                   |
	| Debugging          | Hard              | Easy                   |

	CompletableFuture:
		“Let me run tasks in parallel.”
	
	Structured Concurrency:
		“Let me run tasks in parallel safely and manage them properly.”

225. What are the differences between Synchronized Collection and Concurrent Collection in Java?

		=> while Concurrent Collection is optimized for concurrent access.

	2. => Synchronized Collection uses synchronized keyword to achieve thread-safety, 
	   => while Concurrent Collection uses non-blocking algorithms like CAS (Compare and Swap)

	3. => Synchronized Collection locks the entire collection during modification, leading to potential performance issue	
	   => while Concurrent Collection allows multiple threads to access and modify the collection concurrently

	4. => Examples of Synchronized Collection include synchronizedList, synchronizedSet, synchronizedMap, 
	   => while examples of Concurrent Collection include ConcurrentHashMap, CopyOnWriteArrayList, CopyOnWriteArraySet

230. Two-Phase Commit (2PC) ?
		=> Two-Phase Commit (2PC) is a distributed transaction protocol that ensures all participating systems either commit a transaction together or roll it 			   back together — guaranteeing atomicity across multiple databases or services.

	⚙️ How Two-Phase Commit Works (Step-by-Step)
	🟡 Phase 1: Prepare (Voting Phase)
		Coordinator asks all participants: “Can you commit?”
		Each participant:
			Performs checks
			Locks resources
			Replies YES (ready) or NO (cannot commit)
	
	🟢 Phase 2: Commit / Rollback Phase
		If ALL say YES → Coordinator sends COMMIT

238. Why is circuit breaker not a complete solution ?

			Detects high failure rate
			Opens the circuit
			Stops calling downstream service
			After some time → tries again (half-open state)
			It prevents continuous hammering of a failing service.

	=> ❗ Why Circuit Breaker is NOT a Complete Solution :
		1.It Only Stops Calls — It Doesn’t Fix Root Cause
				If:
					DB is slow
					Thread pool is exhausted
					CPU is 100%
					Circuit breaker does not fix those.
				
				It just says:
					“Stop calling for now.”
					Problem still exists.
		2. It Can Cause Cascading Failures Upstream => No gracefull failure
				If circuit opens:
					Order service fails
					API Gateway fails
					UI fails
					Now user sees failure.
				Without:
					Fallback
					Graceful degradation
					System still breaks.

		3. Wrong Configuration Can Make Things Worse
			If threshold is too low:
					Circuit opens too quickly (false positives)
				If threshold is too high:
					System gets hammered before opening
					So tuning matters.

252. What is Lambda in Java ?
	=> A lambda expression is a way to write an anonymous function (function without name) in a shorter and cleaner way.
		=> Introduced in Java 8

	=> ❌ Problem Before Java 8
		Before Java 8, if you wanted to pass behavior (function) as a parameter, you had to:
			Create an interface
			Implement it
			Create anonymous class
			Override method
		Example (Before Java 8):
			Runnable r = new Runnable() {
			    @Override
			    public void run() {
			        System.out.println("Running");
			    }
			};

	=> With Lambda (After Java 8)
		Runnable r = () -> System.out.println("Running");
		
		🔥 Same logic.
		🔥 70% less code.
		🔥 More readable.

	=> ⚡ Does Lambda Improve Performance?

👉 Short answer: Not directly because of lambda itself.
	Lambda itself does NOT magically make code faster.
		BUT it enables:
			1️⃣ Parallel Streams
		users.parallelStream()
		     .filter(u -> u.isActive())
		     .forEach(System.out::println);
		
		=> Now multiple threads process data.
		So performance improvement comes from:
			Streams
			Parallel processing
			Internal iteration
			Not from lambda alone.

	=> Lambda works only with Functional Interface.
		A functional interface has:
			Exactly one abstract method
		Examples:
			Runnable  -> run()
			Callable  -> call()
			Comparator -> compare()
			Comparable -> compareTo()
			Predicate -> boolean test(T t)
			Function -> R apply(T t)
			Consumer -> void accept(T t)
			Supplier -> T get()

	=> 🎯 What If Interface Has 2 Abstract Methods?  or if it Normal fucntion

			interface Test {
			    void a();
			    void b();
			}
			Then:
				Test t = () -> System.out.println("Hi"); ❌
			Compiler error.
				Because lambda must match exactly one abstract method.

			Why?
			Because:
				Should lambda implement a() or b()?
			
			Ambiguity.
				Compiler cannot decide.

	=> How lambda works internally ?
		=> Lambda expressions in Java are implemented using the invokedynamic instruction introduced in Java 7. 
		   The compiler generates a synthetic private method for the lambda body, and at runtime, the JVM uses LambdaMetafactory to dynamically create an 				   implementation of the functional interface. 
		   Unlike anonymous classes, lambdas do not create separate class files, making them more memory efficient and allowing better JVM optimizations.	

	=> Can Lambda Throw Checked Exception ? : No

			👉 Too many indexes slow down INSERT/UPDATE

257. Why are parallel streams dangerous in Spring Boot ?

	=> Parallel streams use ForkJoinPool.commonPool(), which is shared globally across the JVM. 
	   In a Spring Boot application handling multiple concurrent requests, this can lead to thread contention, 
	   CPU exhaustion, and unpredictable performance. Additionally, parallel streams are not suitable for blocking operations like database or REST calls, 
	   which are common in backend services. 
	   Therefore, it is safer to use a custom ExecutorService or CompletableFuture with a dedicated thread pool 
	   for better control and scalability.

277. What is JPA Specification ?
	=> JPA Specification is a way to build dynamic queries programmatically using the Criteria API.
		It allows you to:

		        return (root, query, criteriaBuilder) ->
		                criteriaBuilder.greaterThan(root.get("age"), age);
		    }
		
		    public static Specification<Employee> hasDepartment(String dept) {
		        return (root, query, criteriaBuilder) ->
		                criteriaBuilder.equal(root.get("department"), dept);
		    }
		
		    public static Specification<Employee> hasSalaryGreaterThan(double salary) {
		        return (root, query, criteriaBuilder) ->
		                criteriaBuilder.greaterThan(root.get("salary"), salary);
		    }
		}

	✅ Step 3: Combine Specifications :

		Specification<Employee> spec = Specification
		        .where(EmployeeSpecification.hasAgeGreaterThan(25))
		        .and(EmployeeSpecification.hasDepartment("IT"))
		        .and(EmployeeSpecification.hasSalaryGreaterThan(50000));
		
		List<Employee> employees = repository.findAll(spec);

		FROM employee
		WHERE age > 25

280. Explain the main method in Java ?

	=> The main() method is the entry point of a Java application.
	   When the JVM starts execution, it looks for the main method and begins running the program from there.

	=> Syntax Breakdown :
	1. public : 
		=> The JVM must be able to access the method from anywhere.
		=> If it is not public: 
				JVM cannot access it → Program will not run.

	2. static :
		=> main() must be static because JVM calls it without creating an object.
		=> if not : new Test().main() -> But JVM does not create objects automatically. So it must be static.

	3. void :
		=> It means main method does not return any value.
		=> The JVM does not expect any return value from the main method.

	4. main :
		=> This is the method name recognized by JVM as the starting point.
		If you rename:
			eg:	public static void start(String[] args)

281. What is a Red-Black Tree ?

		=> A Red-Black Tree is a self-balancing binary search tree where each node is either red or black. 
		   It maintains balance using a set of properties such as the root being black, no two red nodes appearing consecutively, and all root-to-leaf paths 			   having the same number of black nodes. When violations occur during insertion or deletion, the tree uses recoloring and rotations to restore balance. 		   This ensures search, insertion, and deletion operations run in O(log n) time. In Java, Red-Black Trees are used internally in TreeMap, TreeSet, and 		   	   also in HashMap for handling large collision buckets.

284. How to handle concurrent transactions without loosing data ?
		1. Database Transactions :
			Use ACID properties.
					A → Atomicity -> All operations succeed or none
					C → Consistency -> Database remains valid
					I → Isolation -> Transactions do not interfere
					D → Durability -> Changes persist
			=>	In Spring Boot:
						@Transactional
						public void transferMoney(){
						}
			=>	This ensures the transaction is executed safely.

		2. Isolation Levels :
			Isolation levels control how concurrent transactions interact.
				Isolation Level					Problem Prevented
				READ_UNCOMMITTED				nothing
				READ_COMMITTED					dirty reads
				REPEATABLE_READ					non-repeatable reads
				SERIALIZABLE					all concurrency issues
			Example:
				@Transactional(isolation = Isolation.SERIALIZABLE)

	3. Optimistic Locking :
		=>	Assumes conflicts are rare.
				We add a version column.
			Example table:
				id | balance | version
				JPA Entity:
					@Entity
					public class Account {
					    @Id
					    private Long id;
					    private double balance;
					    @Version
					    private Long version;
					}
					
				How it works:
				
					User A reads version = 1
					User B reads version = 1
						User A updates → version becomes 2
						User B tries update → version mismatch ❌
				
				Spring throws:
					OptimisticLockException

		4. Pessimistic Locking :
				Here the database locks the row so other transactions cannot update it.
					Example:
						@Lock(LockModeType.PESSIMISTIC_WRITE)
						Account findById(Long id);
					This ensures:
						Transaction 1 locks row
						Transaction 2 must wait

	=> Use locking, isolation levels, and optimistic concurrency to manage concurrent transactions effectively.
			1. Implement optimistic locking using versioning to prevent lost updates.
			2. Use pessimistic locking to lock records during a transaction.
			3. Set appropriate isolation levels (e.g., READ_COMMITTED) to control visibility of changes.
			4. Utilize database transactions to ensure atomicity and consistency.

287. Difference between Thread vs Runnable.
		=> Both are used to create threads, but they differ in design and flexibility.

		1️⃣ Thread : Thread is a class in Java.
				=> You create a thread by extending the Thread class.
					Example:
						class MyThread extends Thread {
						    public void run() {
						        System.out.println("Thread is running");
						    }
						}
						
						public class Test {
						    public static void main(String[] args) {
						        MyThread t = new MyThread();
						        t.start();
						    }
						}

		2️⃣ Runnable : Runnable is a functional interface.
				You implement it and pass it to a Thread.
			Example
				class MyRunnable implements Runnable {
				    public void run() {
				        System.out.println("Thread is running");
				    }
				}
				
				public class Test {
				    public static void main(String[] args) {
				        Thread t = new Thread(new MyRunnable());
				        t.start();
				    }
				}

	=> Thread is a class used to create threads by extending it and overriding the run() method. 
	   Runnable is an interface that represents a task to be executed by a thread. 
	   Runnable is preferred because it allows multiple inheritance and better resource sharing between threads.

290. Runnable vs Thread — When should we use each ?
		In Java, there are two ways to create a thread:
			1️⃣ Extend java.lang.Thread
			2️⃣ Implement java.lang.Runnable

	1️⃣ Using Runnable Interface (Recommended)
		 Example :
			class MyTask implements Runnable {
			    @Override
			    public void run() {
			        System.out.println("Task running by " + Thread.currentThread().getName());
			    }
			}
			
			public class Main {
			    public static void main(String[] args) {
			        MyTask task = new MyTask();
			        Thread t = new Thread(task);
			        t.start();
			    }
			}

			When should we use Runnable?
				Use Runnable when your class needs to extend another class.
			Example:
				class EmployeeService extends BaseService implements Runnable
				Java does not support multiple inheritance, so if you extend Thread, you cannot extend any other class.
			Advantages
				✔ Better design (separation of task and thread)
				✔ Allows class to extend other classes
				✔ Used with Executor Framework
				✔ Used in thread pools
			
			Example with executor:
				ExecutorService executor = Executors.newFixedThreadPool(5);
				executor.submit(new MyTask());
		=> For modern Java applications, especially multithreaded environments (which you’re interested in), 
		   Runnable is preferred.

	2️⃣ Using Thread Class
			Example
			class MyThread extends Thread {
			    @Override
			    public void run() {
			        System.out.println("Thread running");
			    }
			}
			
			public class Main {
			    public static void main(String[] args) {
			        MyThread t = new MyThread();
			        t.start();
			    }
			}
			When should we use Thread?
			
			Use Thread class when:
				✔ You want to customize thread behavior
				✔ You need to override thread methods
			
			Example:
			class MyThread extends Thread {
			
			    public MyThread() {
			        setName("Worker-Thread");
			    }
			
			    @Override
			    public void run() {
			        System.out.println(getName());
			    }
			}
		=> But this approach is less flexible.

	=> | Feature      | Runnable                 | Thread                      |
	   | ------------ | ------------------------ | --------------------------- |
	   | Type         | Interface                | Class                       |
	   | Inheritance  | Can extend other classes | Cannot extend another class |
	   | Design       | Better separation        | Task tightly coupled        |
	   | Thread Pool  | Supported                | Not recommended             |
	   | Modern usage | Preferred                | Rare                        |

292. Causes of dead lock accurs ? and prevntive methods ?
	=> Deadlock occurs when these 4 conditions exist simultaneously
		1️⃣ Mutual Exclusion – Resource can be used by only one thread at a time
		2️⃣ Hold and Wait – Thread holds one resource and waits for another
		3️⃣ No Preemption – Resource cannot be forcibly taken away
		4️⃣ Circular Wait – Threads wait in a circular dependency
	=> If we break any one of these, deadlock can be prevented.

	==> Runnable : void run();
	
			📌 Key Points
				✔ Does NOT return result
				✔ Cannot throw checked exception
				✔ Used with Thread class or ExecutorService
				✔ Old and simple interface
				
			Eg : class MyTask implements Runnable {
					public void run() {
						System.out.println("Task running...");
					}
				}

				public class Main {
					public static void main(String[] args) {
						Thread t = new Thread(new MyTask());
						t.start();
					}
					}
	==> Callable Interface : V call() throws Exception;
	
			📌 Key Points
				✔ Returns a result
				✔ Can throw checked exception
				✔ Used with ExecutorService
				✔ Introduced in Java 5
				
				
41. ✅ What is Intrinsic Lock in Java?

	==> An intrinis lock is the internal moniter lock that every java object has, 
		which automatically aquired and released when a tread enters and exit syncronized block
		
		EG : 🧠 Key Points
				✔ Every Java object has one intrinsic lock
				✔ Managed automatically by JVM
				✔ Used by synchronized
				✔ Only one thread can hold it at a time
				✔ Ensures mutual exclusion + visibility
				
	==> class Counter {
	
			public synchronized void increment() {
					System.out.println("Thread safe operation");
				}
			}
			
			When thread enters increment():
			👉 Acquires intrinsic lock of Counter object (this)
			👉 Executes method
			👉 Releases lock after completion

42. ✅ What is Race Condition?
	==> A race condtion happens when multiple threads compete to update shared data without proper synchronization,
		causing unpredicatble output
		
	==> to avoid this we can synchronized keyword 
		Eg :
		public void increment() {
			synchronized(this) {
				count++;
			}
		}

=>  What if thenComparing(Employee::getDepartment) returns null?
	What happens?
	How to handle it?

	👉 NullPointerException will occur
	Because internally:
		e1.getDepartment().compareTo(e2.getDepartment())
		And calling compareTo() on null throws NPE.

=> 🔷 How To Handle Null Safely?
		Java provides:
			Comparator.nullsFirst()
			Comparator.nullsLast()
		
	✅ Option 1: Put Nulls First
		List<Employee> sorted = employees.stream()
		    .sorted(Comparator.comparing(Employee::getAge)
		        .thenComparing(
		            Employee::getDepartment,
		            Comparator.nullsFirst(String::compareTo)
		        )
		        .thenComparing(Employee::getSalary))
		    .collect(Collectors.toList());

	=> Now:
			If department is null → comes first
			No exception

33.	What Is Binary Search ? 
	=> Binary Search is an algorithm used to: Efficiently search an element in a sorted array.

	2️⃣ Prerequisite (Very Important)
		👉 Array/List must be sorted
				Ascending or descending — but sorted.
		If not sorted ❌
			Binary search will give wrong result.

=> 	Principle of Binary Search
			Main idea:
				Divide the search space into half every time.
			Steps:
				Find middle element
				Compare target with middle
				If equal → return
				If target < middle → search left half
				If target > middle → search right half
				Repeat
			This is called:
				👉 Divide and Conquer

	=>  int[] arr = {10, 20, 30, 40, 50, 60};
        int target = 50;
        System.out.println(binarySearch(arr, target));+

	=>  public static int binarySearch(int[] arr, int target) {
	        int left = 0;
	        int right = arr.length - 1;
	
	        while (left <= right) {
	
	            int mid = left + (right - left) / 2; // safe way
	
	            if (arr[mid] == target) {
	                return mid;
	            }
	
	            if (arr[mid] < target) {
	                left = mid + 1;
	            } else {
	                right = mid - 1;
	            }
	        }
	
	        return -1; // not found
	    }

        }
